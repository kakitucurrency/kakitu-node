# KAKITU-NODE DEEP TECHNICAL AUDIT
**Date:** 2026-03-28 | **Base:** Nano v25 fork | **Network:** KSHS Live

---

## 1. CRITICAL ISSUES

### 1.1 Node Private Key Exposed via RPC — CRITICAL / 1 line fix
`json_handler.cpp:2926` returns the node identity private key in plaintext:
```cpp
response_l.put ("private", node.node_id.prv.to_string ());
```
Any caller with RPC access can steal the node identity key. Fixed in official Nano v27.1 (PR #4459).
**Fix:** Delete that one line.

---

### 1.2 Unbounded `election_winner_details` Map — CRITICAL
`active_transactions.election_winner_details` is an unbounded `unordered_map` that grows every election win and is only drained when the confirmation height processor catches up. Under spam or high load → memory exhaustion → crash.
Fixed in Nano v27.1 ("Bounding the growth of the election_winner_details set").
**Fix:** Add configurable size cap + eviction when map exceeds limit. Track as metric.

---

### 1.3 No Bounded Block Backlog — CRITICAL
No enforced limit on total unconfirmed transactions across the ledger and AEC. A spam attacker can flood with valid-PoW blocks and exhaust memory.
Nano v28 implemented "Bounded Block Backlog" (BBB) capping at 100,000 unconfirmed tx.
**Fix:** Track total unconfirmed counter. When > configurable max (100k), reject new blocks until backlog drains.

---

### 1.4 Legacy V1 Handshake Still Active — CRITICAL
Both V1 and V2 handshakes are active. `messages.cpp:1471/1514` has TODO comments to remove legacy. V1 does not include genesis block hash in signed data — an adversary can connect to a KSHS node without proving they're on the KSHS network.
**Fix:** Remove V1 handshake. Bump `protocol_version_min` to V2-only peers.

---

### 1.5 Vote Batching Limit of 12 — CRITICAL
`network.hpp:158` — `confirm_ack_hashes_max = 12`. Official Nano v27 raised this to 255.
With limit=12, confirming 10,000 tx requires ~834 vote messages. With 255 it's ~40 — a **20x efficiency gap**. This is the primary reason vote-related spam succeeds on KSHS.
**Fix:** Raise to 255. Update assertion in `messages.cpp:686` and all serialization paths. Requires protocol version bump.

## 2. MISSING POST-v25 NANO FIXES

### 2.1 Fair Queue Processing (Nano v27) — HIGH
Without fair queuing, a single peer can monopolize the block processor queue. Nano v27 added round-robin per-peer sub-queues.
**Fix:** Replace single `std::deque<nano::unchecked_info>` in `blockprocessor.hpp` with per-channel map with round-robin draining.

### 2.2 Legacy Bootstrap Should Be Disabled (Nano v28) — HIGH
`bootstrap_legacy.cpp` and related files still present and active. Nano v28 removed it. It's slower, buggy (race conditions in test comments), and complex.
**Fix:** Set `disable_legacy_bootstrap = true` by default. Plan removal next release.

### 2.3 No Bulk Frontier Scanning for Bootstrap (Nano v26/v27) — HIGH
Ascending bootstrap only samples accounts one-by-one. Nano v26 added bulk frontier scanning (1000 accounts at a time). Cold sync is dramatically faster with it.
**Fix:** Add `frontier_scan` message type to ascending bootstrap protocol.

### 2.4 Cementing Edge Cases (Nano v27.1) — HIGH
FIXME in `core_test/election.cpp:200` ("this test fails on rare occasions") and `test_common/testutil.cpp:83` ("block processor flush is broken") — both signal unresolved cementing race conditions.
**Fix:** Backport v27.1 cementing fixes. Key area: `block_already_cemented_callback` timing race in `active_transactions.cpp:179-183`.

### 2.5 RocksDB Outdated (Nano v28 upgraded to 9.7.2) — HIGH
Bundled RocksDB is pre-9.7.2. TODO in `rocksdb.cpp:107` signals unresolved initialization issues.
**Fix:** Update RocksDB submodule to 9.7.2+. Apply Nano v28 updated `get_db_options()` config.

### 2.6 No Startup Ledger Integrity Verification (Nano v28.2) — HIGH
Node can start with a corrupted ledger and silently operate incorrectly.
**Fix:** On startup verify: (1) sum of balances = genesis supply, (2) rep weights match balance delegation, (3) confirmation heights consistent. Abort if inconsistent.

### 2.7 Incomplete Vote Deduplication (Nano v28) — MEDIUM
No Bloom filter for recently seen vote hashes. Votes are re-broadcast unnecessarily, wasting bandwidth.
**Fix:** Extend existing `publish_filter` (network_filter) in `network.hpp:148` from blocks to votes.

## 3. PERFORMANCE IMPROVEMENTS

### 3.1 Single-Threaded Vote Processor — HIGH
`vote_processor.cpp` runs one thread. Under spam, the queue (capacity 147,456) fills and votes are dropped → delayed confirmation.
**Fix:** Shard by account hash (4 threads, route via `vote->account.bytes[0] % num_threads`). Per-bucket locking on `active_transactions::last_vote`.

### 3.2 Bootstrap Work on Congested Block Processor Thread — HIGH
`bootstrap_ascending/service.cpp:30` has critical TODO: "This is called from a very congested blockprocessor thread. Offload this work to a dedicated processing thread."
**Fix:** Move `block_processor.batch_processed` callback work to bootstrap service thread via lightweight queue.

### 3.3 Additional Priority Bucket (Nano v27) — MEDIUM
Nano v27 added a 63rd bucket for range Ӿ0.000001–Ӿ0.0003 for faucets and minimum-receive. Without it, small-value KSHS transactions (common in retail) compete with large ones.
**Fix:** Add the threshold bucket in `prioritization.cpp`.

### 3.4 Blocking `process_local` Stalls RPC Threads — MEDIUM
`blocking_observer.cpp` blocks the calling thread until a block is processed. Under high RPC load → thread starvation in RPC server.
**Fix:** Convert `process_local` to fully async with promise/future and configurable timeout.

## 4. SECURITY HARDENING

### 4.1 `node_id_private.key` File Permissions Not Restricted — HIGH
File written without explicit permissions (`chmod 0600`). On misconfigured systems it's world-readable (0644).
**Fix:** After write, call `boost::filesystem::permissions(path, owner_read | owner_write)`. Validate on every startup.

### 4.2 HTTP Callback Has No TLS — HIGH
`node.cpp:481-548` uses raw TCP socket. Transaction confirmation data sent over plain HTTP.
**Fix:** Replace with `boost::asio::ssl::stream` for HTTPS endpoints. Add configurable TLS cert verification.

### 4.3 IP-Only Peer Deduplication (No Subnet Limiting for IPv4) — HIGH
`peer_exclusion.cpp` tracks by IP. IPv6 has /64 subnet limiting; IPv4 has none. 3 IPs can run 30 nodes filling 30 slots.
**Fix:** Apply /20 IPv4 subnet limiting. Add per-ASN diversity enforcement. Implement behavior-based reputation scoring.

### 4.4 Genesis Block Not Re-Verified on Startup — MEDIUM
Genesis block signature only verified once at initial ledger init. Subsequent startups trust stored genesis blindly.
**Fix:** On every startup, re-verify genesis block signature against hardcoded public key (O(1) operation).

### 4.5 RPC Has No Rate Limiting — MEDIUM
Unlimited read-only RPCs. Expensive calls (`account_history`, `ledger` full iteration) can exhaust CPU.
**Fix:** Per-IP token bucket rate limiter. Configurable limits per RPC type. Max count params for history endpoints.

## 5. PROTOCOL IMPROVEMENTS

### 5.1 Protocol Version 2 Behind Official (v19 vs v21) — HIGH
`config.hpp:366` — `protocol_version = 0x13` (19). Official Nano is at 21. Min accepted is 0x12 (18) — allows very old vulnerable nodes.
**Fix:** Bump to 0x15 (21), min 0x14 (20). Requires coordinated migration window.

### 5.2 Vote Hinting Rework Incomplete (Nano v26) — HIGH
Nano v26 made major hinted_scheduler improvements for heavy spam resilience. KSHS has the v25-era version.
**Fix:** Backport v26 `hinted_scheduler` and `vote_cache` interaction improvements.

### 5.3 Final Votes Not Priority-Propagated — MEDIUM
Final votes (timestamp=uint64_max) use same flood mechanism as regular votes. No priority path.
**Fix:** Dedicated priority queue for final votes: flood to all PRs immediately, then random subset of non-PRs.

---

## 6. NETWORK/P2P IMPROVEMENTS

### 6.1 No Inbound Traffic Shaping — HIGH
Outbound has `bandwidth_limiter.cpp`. Inbound has no byte-based rate limit. Attacker can saturate receive buffer.
**Fix:** Per-connection inbound byte rate tracking. Back-pressure/close connections that exceed limit.

### 6.2 Single Preconfigured Peer (SPOF) — HIGH
`nodeconfig.cpp:18` — one default peer `peering.kakitu.org`. DNS unavailable or server down = network partition for new nodes.
**Fix:** Define 5+ geographically distributed peers. Comma-separated list. DNS seed mechanism as fallback.

### 6.3 TCP Write Queue Priority is Naive — MEDIUM
`socket.cpp:403` has TODO: "very basic prioritization." Bootstrap responses starved on busy nodes.
**Fix:** Weighted fair queuing: votes/confirm_ack=10, keepalive=5, bootstrap=1. Deficit counter per queue.

### 6.4 No Peer Diversity Enforcement — MEDIUM
No minimum diversity requirement by ASN/region/version. Eclipse attack via single ISP possible.
**Fix:** No more than 20% connections from same /20 subnet. Prefer geographically diverse peers.

---

## 7. RPC/API IMPROVEMENTS

### 7.1 No RPC Versioning — HIGH
TODO in `json_handler.cpp:3304`: "once we get RPC versioning...". Every breaking RPC change silently breaks integrations.
**Fix:** Add `"version": 1` to all responses. Accept `"api_version"` in requests. Plan `/v2/` endpoint for breaking changes.

### 7.2 Deprecated Endpoints Still Fully Active — MEDIUM
`history` action returns `"deprecated": "1"` but still fully works. No sunset pressure on clients.
**Fix:** Add sunset date to deprecated endpoints. After date, return deprecation notice only.

### 7.3 No REST/GraphQL/OpenAPI — LOW
JSON-RPC `{"action": "..."}` over HTTP POST is non-standard. Limits developer adoption.
**Fix:** REST facade in `kakitu-middleware` repo. OpenAPI 3.0 spec generated from RPC definitions.

### 7.4 HTTP Callback Has No Retry/Auth — LOW
Existing callback fires and forgets, no TLS, no auth headers, no per-account filtering.
**Fix:** Proper webhook: HTTPS + configurable auth header + retry with backoff + delivery confirmation.

---

## 8. DATABASE/STORAGE IMPROVEMENTS

### 8.1 LMDB Still Default Backend — HIGH
LMDB: single writer, reader-writer lock contention, scaling degrades with large ledgers.
**Fix:** Make RocksDB default for new installs. Migration tool LMDB→RocksDB.

### 8.2 `backup_before_upgrade` Disabled by Default — MEDIUM
`nodeconfig.hpp:111` — `backup_before_upgrade{ false }`. Crash during DB upgrade = corruption.
**Fix:** Flip to `true`. One-line change, massive safety improvement.

### 8.3 No Database Compaction Scheduling — MEDIUM
RocksDB compaction competes with block processing for I/O at unpredictable times.
**Fix:** Expose `database_compact` RPC. Config for compaction rate limit. Auto-schedule during low-traffic windows.

---

## 9. MONITORING & OBSERVABILITY

### 9.1 No Structured Logging (spdlog) — HIGH
Free-form text logger only. No JSON mode, no log levels beyond time-gate, no log categories.
Nano v27 adopted spdlog. KSHS still on v25 primitive logger.
**Fix:** Integrate spdlog. Structured fields: timestamp, level, component, message, peer_id, block_hash, election_id. JSON output mode.

### 9.2 No Prometheus Metrics Endpoint — HIGH
Stats only via `stats_dump` RPC (requires polling). No `/metrics` HTTP endpoint.
**Fix:** Add Prometheus exposition format HTTP endpoint on configurable port (suggest 44079). Map existing `nano::stats` counters. ~200 lines using existing infra.

### 9.3 No Alerting for Critical States — MEDIUM
No built-in webhook for: vote processor full, block processor stalled, peer count drops, confirmation rate drops.
**Fix:** Configurable alerting webhook that POSTs JSON when key thresholds crossed.

## 11. MOONSHOTS

1. **C++20 Coroutines** — Nano v27 rewrote networking with coroutines. KSHS has zero. 10x connection capacity same CPU.
2. **Formal Verification** — Apply TLA+/Alloy to confirmation height algorithm. Never done for any Nano fork.
3. **Slashing for Equivocating Representatives** — On-chain penalty for double-voting. Makes Byzantine behavior economically irrational.
4. **Multi-Signature Accounts** — Native M-of-N multisig at protocol level. Enables business treasury wallets, escrow.
5. **ZK Proofs for Private Transactions** — Optional zk-SNARK shielded amounts. Private payments on a transparent chain.
6. **Layer-2 Payment Channels** — State channels settling to L1. Streaming payments, micropayments.
7. **Node Clustering / HA** — Active-passive with replicated ledger. Sub-second failover. Essential for institutional operation.
